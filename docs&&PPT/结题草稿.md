# "深海幽灵"——基于端边云协同与多级算力路由的潜艇集群调度系统

## 一、项目背景

### 1.1 技术背景

得益于 DeepSeek-V4-Pro 及 DeepSeek-V4-Flash 的强势发布，凭借其国产化、开源的优势，叠加华为昇腾系列 GPU 算力的大幅跃升，在国有化基础设施上本地化部署世界顶级大模型已具备工程可行性。这为军事领域的智能化决策提供了坚实的算力底座。

### 1.2 场景痛点

在深海作战场景中，潜艇集群、水面边缘航母与陆地计算中心天然构成一个端-边-云分布式系统，但面临以下核心矛盾：

1. **端侧数据积压**：潜艇实时采集大量声呐、水文、态势数据，但受限于深海通信链路的间歇性以及艇内不适宜搭载高成本算力平台，数据只能批量暂存，待上浮与边缘航母建立链路后一次性上传。
2. **边缘算力瓶颈**：航母一次性接收的数据量庞大且突发性强，边缘算力资源有限，无法在时效窗口内消化全部任务。
3. **云端时效不足**：陆地数据中心算力充沛，但受传输延迟与排队处理影响，无法满足战术级实时性要求。

### 1.3 本项目方案

针对上述痛点，本项目提出一种**基于端-边-云协同与多级算力路由**的潜艇集群调度系统：

- **端侧（潜艇）**：作为数据源，间歇式批量上传遥测与态势数据。
- **边缘（航母）**：部署推理速度快的 DeepSeek-V4-Flash 模型（10×H100 算力），负责处理时效性强但复杂度适中的 L1 战术任务。
- **云端（数据中心）**：部署参数量更大的 DeepSeek-V4-Pro 模型（超算级算力），承接复杂度高的 L2 战略任务，以及边缘算力溢出时上抛的 L1 任务。

系统通过**三级评估路由器**智能分流、**动态算力溢出机制**保护边缘节点、**双缓冲读写分离**消除锁竞争，实现了高吞吐、低延迟、强容错的分布式智能决策管线。

## 二、系统架构

### 2.1 端-边-云三层架构总览

本系统采用经典的端-边-云三层分布式架构，各层职责明确、协同配合：

| 层级 | 物理载体 | 算力配置 | 核心职责 |
|------|----------|----------|----------|
| 端侧 (End) | 潜艇集群（16 艘） | 无独立推理算力 | 数据采集与批量上传 |
| 边缘 (Edge) | 水面航母 | 10×H100 80GB GPU | L0 紧急熔断 + L1 战术推理（DeepSeek-V4-Flash） |
| 云端 (Cloud) | 陆地数据中心 | 超算级集群 | L2 战略深度推理（DeepSeek-V4-Pro） |

数据流向：潜艇 → 边缘网关 → 三级路由评估 → L0/L1/L2 分级处理 → 指令下发。

### 2.2 三级评估路由器（Hybrid Evaluator）

路由器是系统的核心大脑，负责将每一条上行数据包分配到最合适的处理层级：

**L0 层——规则引擎前置拦截（硬编码熔断）**

L0 是纯粹的生死攸关层。采用硬编码 if-else 规则，零延迟判定，完全绕过神经网络：

- `emergency == True`：触发紧急中止协议
- `battery_pct < 8%`：强制返航指令
- `hull_pressure_bar > 85 bar`：紧急上浮

设计理念：对于危及人员安全的紧急事件，不允许任何 AI 推理延迟介入，确保毫秒级响应。这是分布式系统中经典的 Circuit Breaker（熔断器）模式。

**L1 层——MLP 神经网络智能分流**

排除 L0 拦截的数据后，系统提取 6 维特征向量（深度、航速、电量、声呐接触数、壳体压力、任务优先级），输入启动时预训练的 MLP 分类器进行二分类：

- 输出 L2 概率 ≥ 0.5：判定为战略级任务，路由至云端
- 输出 L2 概率 < 0.5：判定为战术级任务，进入溢出检查

分类器架构：2 层全连接网络（6 → 32 → 32 → 2），ReLU 激活，Softmax 输出。系统启动时自动生成 4000 条模拟历史数据进行训练（60 epochs，Adam 优化器），训练完成后权重持久化至磁盘，后续启动可直接加载。

**动态算力溢出（Spillover）**

当 MLP 判定为 L1 战术任务时，系统并非直接交由边缘处理，而是额外检查当前 GPU 集群负载：

- **负载 < 85%（正常）**：任务写入边缘节点的双缓冲池，由本地 10×H100 消化
- **负载 ≥ 85%（过载）**：触发算力溢出，任务标记为 `[L1-ESCALATED]`，强制上抛至云端 L2 队列，保护边缘节点不被压垮

这一机制确保了边缘节点在突发流量下的稳定性，是系统弹性伸缩能力的核心体现。

### 2.3 双缓冲机制（Ping-Pong Double Buffering）

双缓冲是解决边缘网关层"高并发写入"与"L1 AI 推理慢读取"之间锁竞争冲突的关键设计。

**问题本质**：网关以 400ms 间隔持续写入新任务，而 L1 推理（调用 DeepSeek-V4-Flash）单次耗时可达数秒。若共用同一数据结构，写入方将被推理阻塞，或推理方读到不一致的中间状态。

**解决方案**：维护两个内存数据库 DB_A 与 DB_B，以及一个写入指针 `active_write_db`：

1. **写入阶段**：新任务始终写入 `active_write_db` 指向的数据库（如 DB_A），写入操作仅需短暂加锁
2. **翻转阶段**：每 3 秒触发一次指针翻转，将写入指向切换至另一个数据库（DB_B），同时取出旧库（DB_A）中积攒的全部任务
3. **处理阶段**：在锁外对取出的任务逐条调用 DeepSeek-V4-Flash 进行 AI 推理，产出战术指令

关键优势：翻转操作仅需极短的临界区（切换指针 + 取出数据），AI 推理在锁外执行，写入方几乎不会被阻塞，实现了真正的读写分离与无锁并发。

### 2.4 GPU 负载模拟与溢出保护

系统模拟 10 张 H100 80GB GPU 的实时运行状态：

- 每张 GPU 维护独立的负载率、VRAM 占用、温度等指标
- 负载以 500ms 为周期随机漂移（±4%），模拟真实工作负载波动
- 系统持续监控 10 张 GPU 的平均负载，一旦超过 85% 阈值即触发溢出保护

溢出保护是系统弹性的最后一道防线：宁可将任务上抛至延迟更高的云端，也不允许边缘节点因过载而崩溃。

### 2.5 关键技术选型详解

#### 2.5.1 MLP 分类器——为什么选择多层感知机？

**解决的问题**：在边缘网关层，每秒有大量遥测数据包涌入，系统需要在极短时间内判定每条数据应交由边缘 L1 处理还是上抛云端 L2 处理。这是一个典型的二分类决策问题。

**选型理由**：

1. **历史数据充足**：潜艇集群长期运行积累了大量标注数据（本系统中模拟生成 4000 条），足以支撑监督学习训练。相比手写规则，数据驱动的方式能捕捉到多维特征间的非线性关系。
2. **推理延迟极低**：MLP 前向传播仅涉及矩阵乘法与激活函数，单次推理耗时在微秒级别（对比 Transformer 的毫秒级），满足网关层对实时性的严苛要求。
3. **模型轻量**：2 层全连接网络参数量仅数千，内存占用可忽略不计，适合部署在资源受限的边缘设备上。
4. **可解释性较好**：相比深层网络，浅层 MLP 的决策边界更易理解和调试，便于在答辩中解释模型行为。

**对比其他方案**：

| 方案 | 推理延迟 | 训练数据需求 | 边缘部署适配性 | 不选择的原因 |
|------|----------|-------------|---------------|-------------|
| 决策树/随机森林 | 极低 | 中等 | 高 | 对连续特征的非线性拟合能力弱于 MLP |
| SVM | 低 | 中等 | 高 | 不易扩展为多分类，概率输出需额外校准 |
| Transformer | 高 | 大量 | 低 | 推理开销过大，不适合网关层实时决策 |
| **MLP（选用）** | **极低** | **中等** | **高** | 兼顾速度、精度与部署便捷性 |

**训练流程**：

1. **数据生成**：系统启动时自动生成 4000 条模拟历史遥测数据，每条包含 6 维特征。标签通过加权打分规则自动标注（优先级权重 0.55 + 声呐接触 0.30 + 深度 0.15 + 电量×优先级交互项 0.10 + 随机噪声），得分 > 0.55 标记为 L2 战略级。
2. **特征标准化**：计算训练集的均值与标准差，推理时对输入做 Z-score 标准化，消除量纲差异。
3. **模型训练**：使用 PyTorch 在 GPU（CUDA）上训练 60 个 epoch，batch size 256，Adam 优化器（lr=3e-3），CrossEntropyLoss 损失函数。
4. **权重持久化**：训练完成后将模型权重、标准化参数保存为 `.pt` 文件，后续启动直接加载，无需重复训练。
5. **在线推理**：接收数据包后提取特征 → Z-score 标准化 → 前向传播 → Softmax 取 L2 概率 → 阈值判定。

#### 2.5.2 双缓冲（Ping-Pong Buffer）——为什么不用消息队列？

**解决的问题**：边缘网关高频写入（400ms/条）与 L1 AI 推理慢速读取（数秒/条）之间的速率失配与锁竞争。

**选型理由**：

1. **极简高效**：仅需两个内存列表 + 一个指针，无需引入 Kafka/RabbitMQ 等重量级中间件，适合边缘设备资源受限的场景。
2. **零拷贝翻转**：指针切换是 O(1) 操作，翻转瞬间完成，临界区极短。
3. **天然批处理**：3 秒积攒窗口内的任务被一次性取出批量处理，减少 AI 调用的上下文切换开销。
4. **确定性延迟**：任务最多等待一个翻转周期（3 秒）即可进入处理，延迟上界可控。

#### 2.5.3 WebSocket——为什么不用 HTTP 轮询？

**解决的问题**：前端大屏需要 500ms 级别的数据刷新频率，展示 GPU 遥测、路由事件等高频变化数据。

**选型理由**：WebSocket 建立持久连接后，服务端可主动推送数据，避免 HTTP 轮询的连接建立开销与无效请求。对于本系统每秒 2 次以上的推送频率，WebSocket 的带宽效率与延迟表现远优于轮询方案。

### 2.6 模拟映射说明——演示环境与真实部署的对应关系

由于实验环境的客观限制，本系统无法真正部署于航母与潜艇之上，也无法处理千万级别的并发数据流。但系统的每一个模块都严格对应真实场景中的一个组件或机制，确保架构设计的有效性可被验证。

| 演示环境中的模拟 | 真实部署中的对应 | 模拟方式 | 验证的核心能力 |
|-----------------|-----------------|----------|--------------|
| 16 艘虚拟潜艇（内存对象） | 物理潜艇集群 | 定时随机生成遥测数据包 | 多源异构数据汇聚与路由分发 |
| 10 个 GPU 负载模拟器 | 10×H100 物理 GPU 集群 | 随机漂移 + 手动 Spike 触发 | 负载感知与弹性溢出保护 |
| 内存字典 DB_A / DB_B | 高性能 KV 存储（如 LevelDB） | Python dict + asyncio Lock | 双缓冲读写分离与指针翻转 |
| DeepSeek API 远程调用 | 本地部署的大模型推理服务 | OpenAI 兼容协议调用云端 API | AI 推理链路的端到端集成 |
| 单机 FastAPI 进程 | 分布式微服务集群 | 协程并发模拟多组件协同 | 异步管道与组件间解耦 |
| 400ms 数据生成间隔 | 毫秒级高并发数据流 | 降频模拟，保留处理逻辑完整性 | 路由决策逻辑的正确性 |
| `/api/spike` 手动触发过载 | 真实突发流量洪峰 | 一键将 GPU 负载拉至阈值以上 | 溢出保护机制的触发与恢复 |

**核心原则**：模拟的是规模与硬件，不模拟的是逻辑与算法。三级路由决策、MLP 分类器训练与推理、双缓冲翻转、溢出判定——这些核心逻辑在演示环境中与真实部署完全一致，仅数据源和硬件层做了等效替代。

### 3.2 后端实现

#### 3.2.1 系统启动流程（Lifespan）

FastAPI 应用通过 `lifespan` 异步上下文管理器实现有序启动：

1. **分类器初始化**：调用 `classifier.load_or_train()`，优先加载磁盘上已有的 `.pt` 权重文件；若不存在则现场生成 4000 条训练数据，在 GPU 上训练 MLP 模型（60 epochs），训练过程在线程池中执行以避免阻塞事件循环。
2. **GPU 监控启动**：`spillover_monitor.start()` 启动 10 个 GPU 单元的负载波动协程。
3. **双缓冲启动**：`ping_pong_db.start()` 启动 3 秒周期的翻转-处理循环。
4. **数据管道启动**：`run_data_pipeline()` 作为后台 asyncio Task 持续运行，驱动整个系统的数据流转。

#### 3.2.2 数据生成与模拟

系统模拟 16 艘潜艇（SUB-001 至 SUB-016）的实时遥测数据，每 400ms 随机选取一艘潜艇生成数据包，包含深度、航速、电量、声呐接触数、壳体压力、任务优先级等字段。每 50 个 tick 强制注入一次紧急事件，确保 L0 熔断路径可被持续验证。

#### 3.2.3 WebSocket 双通道通信

系统设计了两条独立的 WebSocket 通道，分别服务于两个前端视图：

- **`/ws/node`**：每 500ms 推送 GPU 遥测数据（10 张卡的负载/VRAM/温度）与双缓冲状态（当前指针、两库任务数、翻转次数、处理结果）。当发生 L0 熔断或 L1 溢出时，额外推送告警事件。
- **`/ws/center`**：每 400ms 推送路由统计（各级别计数与占比）与最近路由事件流，供全局大屏实时刷新。

此外提供 REST API 辅助端点：
- `GET /api/classifier-report`：返回分类器训练元信息（设备、样本数、损失、准确率）
- `GET /api/sub/{sub_id}/log`：返回指定潜艇的历史遥测与指令日志
- `POST /api/spike`：手动触发 GPU 负载飙升，用于现场演示溢出效果

### 3.3 前端实现

#### 3.3.1 路由与导航

前端基于 React Router v6 实现多视图切换，入口路由配置于 `App.tsx`：

- `/` → 自动重定向至 `/center`
- `/center` → 超算中心全局大屏
- `/node` → 边缘节点监控控制台

顶部导航栏常驻显示系统名称、当前接入的 L1/L2 模型标识、深色/浅色主题切换按钮，以及视图路由切换入口。

#### 3.3.2 边缘节点监控控制台（/node）

采用三栏网格布局，聚焦于硬件压榨与双缓冲流转的微观视角：

**左栏——硬件遥测面板（Hardware Telemetry）**

以 10 个独立进度条可视化 10 张 H100 GPU 的实时状态，每张卡显示负载百分比、VRAM 占用与温度。进度条颜色随负载动态变化：绿色（< 65%）→ 黄色（65%-85%）→ 红色（> 85%）。顶部大字号显示集群平均负载，过载时切换为红色并标注"严重过载"。

**中栏——双缓冲引擎视图（Buffer Core）**

通过 `DbBlock` 组件动态展示 DB_A 与 DB_B 的实时状态，每个数据库块标注当前角色（[写入中] / [处理中] / [空闲]），并显示翻转次数、当前写入/读取指向。下方滚动列表展示最近处理完成的任务及其 AI 产出的战术指令。

**右栏——算力溢出监控（Spillover Alert）**

当集群平均负载超过 85% 阈值时，顶部显示红色告警框，标注当前负载值与阈值。下方展示吞吐量统计（翻转次数、排队任务数）以及实时告警日志流，记录每一次溢出事件的时间戳、源潜艇编号与上抛原因。

#### 3.3.3 超算中心全局大屏（/center）

同样采用三栏布局，聚焦于全局算力路由与集群态势的宏观视角：

**左栏——潜艇集群阵列（Swarm Map）**

以 4×4 网格展示 16 艘潜艇的实时状态。每个格子根据最近一次路由结果着色：L0 红色闪烁（紧急）、L1 绿色（正常边缘处理）、L1-ESCALATED 橙色（溢出上抛）、L2 蓝色（战略任务）。点击任意潜艇可弹出详情模态框，通过 REST API 获取该艇的完整遥测与指令历史。

**中栏——三级路由大脑（Global Router）**

顶部展示分类器训练铭牌，包含训练设备（CPU/CUDA）、样本数、训练轮次、最终损失值与准确率，证明模型确实经过真实训练。下方以四条统计条分别展示 L0、L1、L1-ESCALATED、L2 四个路由通道的累计计数、占比与进度条。底部为实时路由事件流，滚动展示最近 10 条路由决策记录。

**右栏——超算深加工日志（Cloud Processing Log）**

#### 3.3.4 实时通信机制

前端通过自定义 `useWebSocket<T>(path)` Hook 管理 WebSocket 连接：

- 自动连接至后端对应通道（`ws://host:8000/ws/node` 或 `/ws/center`）
- 断线后 2 秒自动重连，保证长时间运行的稳定性
- 返回最新数据帧与连接状态，组件可直接消费

对于低频数据（如分类器报告），使用 `useApi<T>(path, pollMs)` Hook 进行 REST 轮询（默认 2 秒间隔）。

#### 3.3.5 UI 设计风格

整体采用军工级深色大屏控制台风格，核心设计原则：

- **深色主题为主**：以深灰/黑色为底，降低视觉疲劳，突出数据信息
- **等宽字体**：所有数据展示使用 monospace 字体，营造终端/指挥舱氛围
- **大字号数据**：关键指标（负载率、路由计数）使用大号字体，确保远距离可读
- **色彩语义化**：绿色=正常、黄色=警告、红色=危险/紧急、蓝色=战略级，与路由层级一一对应
- **去 AI 化**：不使用任何闪烁星星、机器人图标等幼稚元素，保持工业级严肃感
- **CSS 变量驱动主题**：通过 CSS 自定义属性实现深色/浅色一键切换，偏好持久化至 localStorage

## 四、系统亮点与创新点

1. **真实 AI 推理链路**：非纯 Mock 演示，系统实际调用 DeepSeek 大模型完成战术/战略推理，路由分类器在启动时真实训练。
2. **多级弹性调度**：L0 硬熔断 → L1 神经网络分流 → 动态溢出上抛，三道防线层层递进，兼顾实时性与可靠性。
3. **双缓冲无锁并发**：借鉴图形学 Ping-Pong Buffer 思想，以极低的锁开销实现高并发写入与慢速 AI 推理的解耦。
4. **可观测性**：双通道 WebSocket 实时推送，前端多视角大屏全方位展示系统内部状态，便于答辩现场演示与问题定位。
5. **演示友好**：提供 `/api/spike` 一键触发 GPU 过载，可在答辩现场实时演示溢出保护机制的触发与恢复全过程。
